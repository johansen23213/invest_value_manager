# ValueHunter v1.0 — Migration Guide

> **TL;DR:** No file moves. No data migration. Existing workflow (`python3 tools/session_opener.py`, thesis markdown, decisions_log.yaml) continues to work unchanged. The v1.0 architecture adds new layers **alongside** the existing system.

## Guiding principle

**Evolve in place.** Every existing tool, skill, state file, thesis, sector view, principle, and decision remains exactly where it is. Sprint 1 adds schemas and the orchestrator skeleton; Sprints 2-4 add agents, scrapers, scheduler, dashboard. At no point are existing files rewritten except `CLAUDE.md` (evolved, principles preserved verbatim).

## Rollback

Every new artefact lives under a new directory. Rollback = `rm -rf orchestrator/ agents/ models/ scrapers/ scheduler/ notifications/ dashboard/ knowledge_base/ tests/ docs/ARCHITECTURE.md docs/MIGRATION.md pyproject.toml` and revert `CLAUDE.md` from git. Existing v4.6 system is untouched.

## Backward-compatibility tests

`tests/test_regression.py` (Sprint 1) asserts:
1. `python3 tools/session_opener.py` runs without error.
2. `python3 tools/portfolio_stats.py` produces expected shape.
3. `python3 tools/command_center.py --fresh` generates HTML.
4. All existing YAML files parse into schemas without violation (or violations are documented).

## What is **not** migrated

- Individual thesis markdown → JSON conversion is **not** a migration. Markdown remains the source of truth; JSON is a cached structured view.
- Existing HTML dashboards (`command_center.html`, `ownership_graph.html`, `risk_heatmap.html`, `portfolio_dashboard.png`) are **not** deprecated. Streamlit is a companion.
- `.claude/skills/` structure is **not** refactored. The 24 skills remain as-is.
